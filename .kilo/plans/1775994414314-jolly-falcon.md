## RigSense Phase 2 Plan

### Recommendation

Phase 2 should focus on making the current product genuinely useful before expanding breadth. The strongest sequence is:

1. Real pricing and retailer freshness
2. Recommendation and comparison UX
3. Benchmark depth and side-by-side decision support
4. Forum threading and moderation
5. Admin CMS and ingestion tooling

This order builds on what already exists in `README.md`, `prisma/schema.prisma`, `src/app/page.tsx`, and `src/components/builder/builder-workbench.tsx`: a working catalog, builder, saved builds, public content, and forum. The biggest gap is that the product still relies on seeded data and limited decision support, so Phase 2 should make the catalog and builder more trustworthy first.

### Phase 2A - Commerce-Grade Catalog

Goal: turn the seeded parts catalog into a living source of truth.

- Add pricing provider abstractions for API-backed sources and manual fallback
- Store retailer/source metadata, refresh timestamps, and stale-price states using existing `Part.priceSource` and `Part.lastUpdated`
- Build scheduled refresh jobs and a failure/retry strategy
- Show price freshness, source labels, and stale indicators on parts, builder, and public build pages
- Add history-ready schema support if price trend charts are desired soon after launch

Deliverables:

- provider layer in `src/lib/` for pricing integrations
- background refresh workflow and admin/manual refresh trigger
- Prisma migration if price history or retailer offers are added
- UI updates across part cards, part detail, and builder totals

### Phase 2B - Recommendation Engine

Goal: help users answer "what should I buy next" instead of only showing raw catalog data.

- Introduce use cases such as budget gaming, creator workstation, quiet build, and upgrade path recommendations
- Generate recommendations from compatibility data, price bands, benchmark signals, and saved/trending builds
- Add comparison surfaces: recommended alternatives, value picks, and upgrade suggestions
- Expose recommendation modules on homepage, category pages, part detail pages, and builder empty states

Deliverables:

- recommendation service with deterministic scoring first, ML later if needed
- budget/performance profile inputs in builder flow
- reusable recommendation card and comparison table components
- analytics events to measure recommendation CTR and builder conversion

### Phase 2C - Benchmark and Comparison UX

Goal: make benchmark data actionable in the buying journey.

- Expand `Benchmark` usage into normalized comparison views for parts and full builds
- Add side-by-side compare pages for CPUs, GPUs, and completed builds
- Surface workload-oriented summaries such as 1080p esports, 1440p AAA, rendering, and efficiency
- Tie benchmark insights directly into recommendations and builder warnings

Deliverables:

- compare route(s) and compare state management
- normalized benchmark presentation components
- benchmark-to-part/build linking improvements
- seed/data import format for future automated ingestion

### Phase 2D - Community Depth

Goal: move from simple Q&A to a durable knowledge layer.

- Extend forum model from single-level answers to threaded replies
- Add moderation tools: report, hide, lock, review queue, and author/admin actions
- Improve trust features: badges, answer quality indicators, edit history, and rate limits
- Connect guides, benchmarks, and builds into forum discussions to reduce content silos

Deliverables:

- Prisma changes for threaded discussions and moderation entities
- protected moderation routes and server actions
- anti-abuse guardrails and audit trails

### Phase 2E - Admin CMS and Content Ops

Goal: reduce developer dependency for content and catalog operations.

- Create admin CRUD for guides, benchmark entries, featured parts, and moderation queues
- Add draft/publish workflows for editorial content
- Support manual catalog corrections and benchmark curation
- Prepare ingestion hooks for later automated benchmark/pricing feeds

Deliverables:

- role-based admin area
- content editing forms with validation
- audit-friendly publish and rollback strategy

### Technical Cross-Cutting Work

- Add background job support for pricing refresh and future ingestion
- Add observability for provider failures, stale data, and recommendation quality
- Expand test coverage around compatibility, pricing transforms, recommendation scoring, and forum permissions
- Revisit caching strategy for public catalog and benchmark pages once external data is live
- Keep Next.js 16 patterns aligned with the local docs in `node_modules/next/dist/docs/` before implementation

### Suggested Milestones

#### Milestone 1

External pricing with freshness UI, plus schema/supporting jobs.

Success criteria:

- Pricing provider abstraction supports at least one external source and one manual fallback path
- Parts show source, freshness timestamp, and stale state in catalog, part detail, and builder surfaces
- Refresh job flow exists with logging, retry behavior, and a manual trigger path
- Data model is ready for current pricing needs and optional future history/offer expansion

#### Milestone 2

Recommendation engine for homepage, part detail, and builder workflow.

Success criteria:

- Deterministic recommendation scoring exists for at least budget, performance, and upgrade use cases
- Recommendations appear on homepage, part detail, and builder contexts
- Builder can suggest alternatives or next-best options when a part is missing, overpriced, or incompatible
- Recommendation events are measurable so quality can be tuned later

#### Milestone 3

Benchmark compare experience and deeper decision-support surfaces.

Success criteria:

- Users can compare CPUs, GPUs, or completed builds side by side
- Benchmark views are normalized by workload and easy to scan
- Recommendation and builder flows can reference benchmark insights
- Benchmark data format is ready for future ingestion beyond seeded entries

#### Milestone 4

Threaded forum plus moderation foundation.

Success criteria:

- Forum supports replies beyond the current flat answer model
- Moderation actions exist for report, hide, lock, and review workflows
- Basic trust and abuse controls are in place
- Guides, builds, or benchmark content can be referenced from discussions

#### Milestone 5

Admin CMS for guides, benchmarks, and catalog operations.

Success criteria:

- Admin users can create, edit, draft, publish, and roll back content safely
- Catalog and benchmark corrections can be handled without code changes
- Moderation queues are available in the admin surface
- Content operations are auditable and support future ingestion hooks

### Why This Sequence

- The builder and catalog already exist, so pricing/recommendation work compounds immediately
- Benchmarks become more valuable once recommendations can cite them
- Forum/admin work matters, but they do not improve the core build-selection loop as directly as pricing and recommendations
- This sequence reduces the risk of building moderation and CMS tooling around content/data flows that are still immature

### Execution Advice

If only one Phase 2 initiative is started now, choose pricing + recommendation foundations together. That creates the highest user-visible value and unlocks most later roadmap items.
